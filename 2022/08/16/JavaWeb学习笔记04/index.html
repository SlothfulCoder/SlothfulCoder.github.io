JavaWeb学习笔记04

发表于 2022-08-16 更新于 2021-09-14

关键字：会话技术、cookie、session、jsp

1.会话技术

会话：浏览器第一次给服务器资源发送请求，会话建立，直到有一方断开为止，一次会话中包含多次请求和响应

功能：在一次会话的范围内的多次请求间，共享数据

方式：

  1. 客户端会话技术：Cookie
  2. 服务器会话技术：Session

2.Cookie

Cookie是一种客户端会话技术，将数据保存到客户端，用于实现一次会话多次请求间的通信。

由服务器创建Cookie，把Cookie封装到响应头Set-Cookie中发送给客服端，客户端再次向发送服务器请求携带cookie数据，服务器就能在这次请求得到上次请求的数据了

方法

  1. new Cookie(String name, String value) ：创建Cookie对象，绑定数据

  2. response.addCookie(Cookie cookie) ：把cookie设置到响应头

  3. Cookie[] request.getCookies() ：获取Cookie，拿到数据

  4. cookie.setMaxAge(int seconds)：设置cookie的存活时间

    • 正数：将cookie数据写到硬盘文件中，持久化存储。并把参数作为cookie存活的秒数，时间过短可能在会话结束前就销毁
    • 负数：默认值，会话结束，cookie自动销毁
    • 零：销毁存活时间未到的cookie

  5. cookie.setPath(String path)：设置cookie的获取范围。默认值是当前的虚拟目录，设置为”/“代表部署到tomcat下的所有web项目共享

  6. cookie.setDomain(String path)：如果设置一级域名相同，那么多个服务器之间cookie可以共享

cookie细节

  1. 一次可以发送多个cookie，只需要创建多个Cookie对象，用response调用多次addCookie方法即可
  2. 默认情况下，浏览器关闭后，cookie数据被销毁（会话结束）
  3. 在tomcat 8 之前 cookie中不能直接存储中文数据，需要将中文数据转码—一般采用URL编码(%E3)
  4. 在tomcat 8 之后，cookie支持中文数据。特殊字符还是不支持，建议使用URL编码存储，URL解码解析
    • URLEncoder.encode(String s, String enc)
    • URLDecoder.decode(String s, String enc)
  5. 在一个tomcat服务器中，部署多个web项目，默认情况下这些web项目的cookie不能共享，可以设置cookie.setPath(String path)为”/“，使这些web项目共享这条cookie数据
  6. 默认情况下，不同tomcat服务器的cookie不能共享，想要共享需要setDomain方法设置

特点

  1. cookie存储数据在客户端浏览器
  2. 浏览器对于单个cookie 的大小有限制(一般4kb左右) 以及 对同一个域名下的总cookie数量也有限制(一般20个以内)

作用

  1. 客户端安全级别较低，cookie一般用于存出少量的不太敏感的数据
  2. 在不登录的情况下，完成服务器对客户端的身份识别

3.Session

Session使一种服务器端会话技术，在一次会话的多次请求间共享数据，将数据保存在服务器端的对象中

Session是依赖于cookie的，为了保证每次客户端访问能够找到自己的session数据，服务器在保存session数据的同时，为session生成一个唯一的id，并把这个id以cookie的形式传给客户端，客户端下次请求把cookie再传回来，服务器就可以在新的请求找到客户端的session数据了

方法

  1. HttpSession session = request.getSession()：获取session对象
  2. void setAttribute(String name, Object value)：设置session数据
  3. Object getAttribute(String name) ：获取session数据
  4. void removeAttribute(String name)：删除session数据

细节

  1. 默认情况下，客户端关闭，就不能获取到session的数据了，因为会话结束，cookie就销毁了，就不能再向服务器发送session的id

  1. 默认情况下，服务器关闭，就不能获取到session的数据了，因为session数据保存在服务器的内存中，服务器关闭，内存中的数据会被销毁。想要持久化保存session，需要对session钝化、再活化

    • 钝化：服务器正常关闭之前，将session对象序列化到硬盘上
    • 活化：在启功服务器后，将session文件转化为内存中的session对象即可，这样得到的session对象能保证数据不变，但是内存地址不一样了

    钝化的session文件在work目录下，tomcat正常关闭和启动时，会自动帮我们完成钝化和活化，使用集成到idea的tomcat也能在生成的配置文件目录的我认得文件夹下完成钝化，但是无法正确的活化，活化时会先删除掉这个我work文件夹，重新创建一个

  2. session销毁时机

    • 服务器关闭

    • session对象调用invalidate()方法

    • session默认失效时间：30分钟

  1. session可以存储任意类型、任意大小的数据

session和cookie的区别

  1. session存储数据在服务器端，cookie在客户端
  2. session没有数据大小限制，cookie有
  3. session数据安全，cookie相对于不安全

4.JSP入门

JSP： Java Sever Pages，Java服务器端页面，可以理解为一个特殊页面，其中既可以指定定义html标签，又可以定义java代码

jsp本质上是一个Sevlet，tomcat会将jsp文件转化为一个java文件，保存到生成的配置文件目录的work文件夹下，并将其编译成class文件，这个自动生成的类继承HttpJspBase类，而HttpJspBase类实现了Servlet

jsp脚本

jsp会转换成一个servlet类，因此也有service方法

  1. <% 代码 %>：定义java代码，在service方法中，service方法中可以定义什么，该脚本就可以定义什么
  2. <%! 代码 %>：定义的java代码，在jsp转换后的java类的成员位置
  3. <%= 代码 %>：定义的java代码，会输出到页面上，输出语句可以定义什么，该脚本就可以定义什么

jsp内置对象

这是在jsp页面中不需要获取和创建，可以直接使用的对像，一共有9个

  1. request
  2. response
  3. out：字符输出流对象，可以将数据输出到页面上，和response.getWriter()类似，但是也存在不同
    • response.getWriter()数据输出永远在out.write()之前
    • 因为在tomcat服务器真正给客户端做出相应之前，会先找response缓冲区数据，再找out缓冲区数据
